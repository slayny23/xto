# ПР №3. Права доступа Linux и управление пользователями

## 1. Пользователи и группы

Созданные пользователи и их роли:

| Пользователь | Группа | Роль |
|---|---|---|
| alice | developers | Администратор проекта |
| bob | developers | Разработчик |
| carol | auditors | Аудитор |

Поля /etc/passwd (разбор строки alice):

`alice:x:1001:1003::/home/alice:/bin/bash`

1. `alice` — имя пользователя
2. `x` — пароль хранится в /etc/shadow
3. `1001` — UID (идентификатор пользователя)
4. `1003` — GID (идентификатор основной группы)
5. `` — поле комментария (пусто)
6. `/home/alice` — домашняя директория
7. `/bin/bash` — командная оболочка

### /etc/shadow

Файл хранит хэши паролей, дату последней смены пароля и политику устаревания.
Недоступен обычным пользователям (права `-rw-r-----`) — чтобы никто не мог
получить хэш и взломать пароль офлайн-атакой.

## 2. Права доступа chmod/chown

Итоговые права /srv/project/code: `chmod 770`, владелец `alice:developers`

- bob (в группе developers): **не мог** создать файл при 750, потому что группе
  дано только чтение и выполнение (r-x), запись отсутствует. После chmod 770 —
  смог, так как группа получила право записи (rwx).
- carol (не в группе): **не могла** войти в директорию, потому что для остальных
  (other) установлены права `0` — полный запрет.

### Файл q1.txt

Права `640` = `-rw-r-----`: владелец alice читает и пишет, группа auditors —
только читает, остальные — нет. Carol как член группы auditors может читать файл.

## 3. ACL

Задача: дать carol доступ на чтение без изменения группы файла.

Команда: `sudo setfacl -m u:carol:r-x /srv/project/code`

Вывод getfacl после изменения:
# file: srv/project/code
# owner: alice
# group: developers
user::rwx
user:carol:r-x
group::rwx
mask::rwx
other::---

ACL лучше стандартной модели тем, что позволяет назначить права конкретному
пользователю (carol) не меняя группу файла и не добавляя carol в группу developers.

## 4. sudo-политики

| Пользователь | Разрешено | Ограничение |
|---|---|---|
| alice | Всё (NOPASSWD) | Нет |
| bob | apt, apt-get | Только пакетный менеджер |
| carol | journalctl, cat /var/log/* | Только просмотр логов |

alice использует NOPASSWD так как является администратором.
bob и carol получили минимально необходимые права — принцип наименьших привилегий (УПД по Приказу ФСТЭК №17).

## 5. PAM

Модуль pam_unix.so отвечает за проверку пароля через /etc/shadow.

required означает: модуль обязателен — если вернёт ошибку, аутентификация отклонена.

Чтобы добавить минимальную длину пароля 12 символов:
password requisite pam_pwquality.so minlen=12

## Выводы

Стандартная модель прав Linux (chmod/chown) разграничивает доступ по принципу
владелец/группа/остальные. ACL расширяет модель для конкретных пользователей.
sudo реализует принцип наименьших привилегий. PAM обеспечивает гибкую аутентификацию.
Все механизмы соответствуют группе мер УПД Приказа ФСТЭК №17.
