# ПР №6. Анализ сетевого трафика: tcpdump и Wireshark

## 1. Разбор строки tcpdump

Строка: `11:09:21.781143 IP 127.0.0.1.60110 > 127.0.0.1.8080: Flags [S], seq 1846055698, length 0`

| Поле | Значение | Что означает |
|------|----------|--------------|
| `11:09:21.781143` | Время | Метка времени с микросекундами |
| `IP` | Протокол | Сетевой уровень IPv4 |
| `127.0.0.1.60110` | Источник | IP и порт отправителя |
| `127.0.0.1.8080` | Назначение | IP и порт получателя |
| `Flags [S]` | TCP-флаг | SYN — начало TCP-соединения |
| `length 0` | Длина | Нет полезной нагрузки (SYN не несёт данных) |

## 2. Перехваченные данные HTTP

![Пароль в открытом виде](screens/http-password.png)

Что видно в трафике: `username=slayny23&password=xx1234`, заголовок `Authorization: Basic c2xheW55MjM6eHgxMjM0`

Кто в реальной сети может так же прочитать этот трафик: любой участник сети — провайдер, системный администратор, злоумышленник проводящий ARP-spoofing атаку

## 3. Follow TCP Stream

![Follow TCP Stream](screens/follow-stream.png)

Что видит злоумышленник перехвативший этот трафик: полный HTTP-диалог — все заголовки запроса, логин, пароль в открытом виде, тело ответа сервера

## 4. HTTP vs HTTPS — сравнение

| Что наблюдаем | HTTP | HTTPS |
|---|---|---|
| IP-адреса | Видны | Видны |
| Заголовки | Видны | Не видны (зашифрованы) |
| Тело запроса | Видно | Не видно (зашифровано) |
| Логин/пароль | Видны | Не видны |
| Имя домена | Видно в Host: | Видно в TLS SNI |

## 5. DNS и приватность

Какие домены видны в DNS даже при HTTPS: github.com, example.com, google.com — все запрошенные домены видны в открытом виде в DNS-запросах

Что это означает для приватности: провайдер и злоумышленник в сети знают какие сайты посещает пользователь даже если содержимое зашифровано. Решение — DoH (DNS over HTTPS) или DoT (DNS over TLS)

## 6. Base64

Декодированная строка: `c2xheW55MjM6eHgxMjM0` → `slayny23:xx1234`

Почему Base64 не является шифрованием: Base64 — это кодирование для передачи бинарных данных в текстовом виде. Не требует ключа, декодируется любым инструментом за долю секунды командой `base64 -d`. Любой кто перехватил трафик мгновенно получает логин и пароль.

## 7. tshark vs tcpdump

| Инструмент | Когда использовать |
|---|---|
| tcpdump | Быстрый захват на сервере без GUI, минимум зависимостей, написание скриптов |
| tshark | Продвинутые display-фильтры, статистика протоколов, автоматизация анализа |
| Wireshark | Интерактивный анализ с GUI, Follow TCP Stream, визуальная работа с pcap |

## 8. Каналы утечки через сеть

Какие меры защиты закрывают сетевой канал утечки:
- HTTPS — шифрует содержимое HTTP-трафика и заголовки
- DoH/DoT — шифрует DNS-запросы, скрывает посещаемые домены
- VPN — шифрует весь трафик включая метаданные
- TLS mutual authentication — защищает от MITM-атак

## Выводы
- HTTP передаёт все данные в открытом виде — пароли перехватываются тривиально
- HTTPS шифрует содержимое, но DNS-запросы остаются открытыми (утечка метаданных)
- Base64 не является защитой — это кодирование, не шифрование
- tcpdump/tshark/Wireshark позволяют полностью восстановить HTTP-сессию
- Для полной защиты необходим HTTPS + DoH/DoT

## 10. Контрольные вопросы

**1. Что такое promiscuous mode и зачем он нужен снифферу?**

Promiscuous mode — режим сетевого адаптера при котором он принимает все пакеты проходящие через сеть, а не только адресованные непосредственно ему. Без этого режима сниффер видит только свой трафик. Сниффер включает promiscuous mode чтобы захватывать трафик других хостов в сети.

**2. Чем отличается capture filter от display filter в Wireshark?**

Capture filter (синтаксис BPF, как в tcpdump) применяется на этапе захвата — отфильтровывает пакеты до записи, уменьшает объём данных и нагрузку. Display filter (синтаксис Wireshark) применяется к уже захваченным пакетам при просмотре — не удаляет пакеты, только скрывает их. Capture filter нельзя изменить без перезапуска захвата, display filter меняется в реальном времени.

**3. Что такое Follow TCP Stream и для чего используется?**

Follow TCP Stream — функция Wireshark которая собирает все пакеты одного TCP-соединения и отображает полный диалог клиент-сервер в читаемом виде (синим — запрос клиента, красным — ответ сервера). Используется для восстановления HTTP-сессий, просмотра передаваемых данных, анализа атак и отладки сетевых приложений.

**4. Почему Base64 в HTTP Basic Auth не является защитой? Что нужно использовать вместо этого?**

Base64 — это кодирование, а не шифрование. Оно не требует ключа и декодируется мгновенно любым инструментом (`base64 -d`). Строка `c2xheW55MjM6eHgxMjM0` моментально превращается в `slayny23:xx1234`. Вместо этого нужно использовать HTTPS (TLS) который шифрует весь трафик включая заголовок Authorization, или современные методы аутентификации — OAuth 2.0, JWT-токены, API-ключи.

**5. Какие данные утекают даже при использовании HTTPS? Как это исправить?**

При HTTPS утекают: DNS-запросы (имена всех посещаемых доменов), IP-адреса источника и назначения, TLS SNI (имя хоста в момент установки соединения), объём и тайминг запросов (metadata). Исправить: DNS-утечку закрывает DoH или DoT; SNI-утечку закрывает ECH (Encrypted Client Hello); IP-адреса скрывает VPN или Tor.

**6. Сисадмин говорит: у нас свитч, а не хаб — значит сниффинг невозможен. Он прав?**

Нет, сисадмин неправ. Свитч действительно защищает от пассивного сниффинга — он отправляет пакеты только на нужный порт, а не всем. Однако существуют методы обхода: ARP-spoofing (злоумышленник подменяет ARP-таблицы и перенаправляет трафик через себя), MAC flooding (переполнение таблицы MAC-адресов переводит свитч в режим хаба), SPAN-порты (зеркалирование трафика администратором). Свитч снижает риск но не исключает его.
