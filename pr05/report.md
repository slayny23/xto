## 2. Шифрование

Зашифрованный файл (первые 5 строк):

----BEGIN PGP MESSAGE-----
hQIMA09Y8azKC+oOAQ/8CFM1rxllpNHCt0qFGj5WXNGqWqYvQaIc0S6bJbud13p9

L6Nhb7pJsQkodv1f/wJJHL4+dY2jSb/HX9fgMxosvweGiml86GDQuBCzoliBCXX2

...

Можно ли прочитать исходные данные из зашифрованного файла: нет, данные
полностью нечитаемы без приватного ключа получателя.

Что произойдёт если ввести неверный пароль при расшифровке: GPG вернёт
ошибку "decryption failed" и не покажет содержимое файла.

Почему нельзя расшифровать файл зашифрованный для другого пользователя:
файл зашифрован публичным ключом получателя, расшифровать его может только
владелец соответствующего приватного ключа.

## 3. Цифровая подпись

Вывод gpg --verify (Good signature):

gpg: Подпись сделана Пт 19 июн 2026 10:17:43 +05

gpg: ключом RSA с идентификатором 14A2F6B3A6066EF57AE7DD6F1DC30FB7C12167F8

gpg: Действительная подпись пользователя "slayny23 park243@mail.ru"

Вывод gpg --verify после изменения файла (BAD signature):

gpg: Подпись сделана Пт 19 июн 2026 10:17:43 +05

gpg: ключом RSA с идентификатором 14A2F6B3A6066EF57AE7DD6F1DC30FB7C12167F8

gpg: ПЛОХАЯ подпись пользователя "slayny23 park243@mail.ru"

Почему подпись перестала быть действительной: подпись — это хэш исходного
файла, зашифрованный приватным ключом. После изменения файла его хэш стал
другим и не совпадает с подписью.

Что именно проверяет gpg --verify: вычисляет хэш файла и сравнивает его
с хэшем из файла подписи, расшифрованным публичным ключом подписанта.

Зачем проверять подпись локально если GitHub уже показывает Verified:
GitHub проверяет только по ключам добавленным в аккаунт. Локальная проверка
независима от GitHub и подтверждает подлинность коммита без доверия к
третьей стороне. Это важно при работе с чужими репозиториями.

## 5. rm vs shred

| Операция | Что происходит с данными на диске | Можно восстановить? |
|----------|----------------------------------|---------------------|
| rm | Удаляется только запись в метаданных (inode), блоки данных остаются | Да, пока блоки не перезаписаны |
| shred | Содержимое блоков перезаписывается случайными данными 3+ раз | Нет |

Почему данные остаются после rm: rm удаляет только ссылку на файл в
таблице inode. Сами блоки данных на диске остаются нетронутыми до тех
пор, пока файловая система не выделит их под новый файл.

Сценарий когда нужно затирать свободное место: перед продажей или передачей
жёсткого диска, когда на нём ранее хранились конфиденциальные данные
(пароли, документы, ключи), удалённые обычным rm.

## Выводы

GPG обеспечивает два вида защиты: шифрование (конфиденциальность) и
цифровую подпись (целостность и подлинность). Подпись коммитов позволяет
доказать что коммит сделан именно владельцем ключа — GitHub подтверждает
это бейджем Verified. shred в отличие от rm реально уничтожает данные,
перезаписывая блоки диска. Все инструменты соответствуют разделу 1.3
«Криптозащита» программы курса.
