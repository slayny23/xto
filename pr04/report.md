# ПР №4. Аудит событий: journalctl и auditd

## 1. journalctl — системный журнал

### Что зафиксировал журнал о команде sudo cat /etc/shadow

июн 17 16:52:07 sll sudo[3047]: ssl : TTY=pts/1 ; PWD=/home/ssl/xto ; USER=root ; COMMAND=/usr/bin/grep -E alice|bob|carol /etc/shadow

Разбор полей: дата и время события, имя хоста (sll), процесс (sudo[3047]),
пользователь (ssl), терминал (pts/1), рабочая директория, от чьего имени
выполнена команда (root), и сама команда.

### Ошибки в системе с последней загрузки

Обнаружены 3 типа ошибок:
- Failed to start app-gnome-gnome\x2dkeyring (2 ошибки GNOME keyring)
- 3 неудачных попытки ввода пароля sudo пользователем ssl

### Статистика входов

- Успешных входов SSH: 0 (SSH не использовался)
- Неудачных попыток: 3 (неверный пароль sudo)

## 2. auditd — настройка правил

### Применённые правила (вывод auditctl -l)

-w /etc/passwd -p rwa -k auth-files

-w /etc/shadow -p rwa -k auth-files

-w /etc/group -p rwa -k auth-files

-w /etc/sudoers -p rwa -k priv-files

-w /etc/sudoers.d -p rwa -k priv-files

-w /etc/ssh/sshd_config -p rwa -k ssh-config

-a always,exit -F arch=b64 -S open,openat -F success=0 -F key=access-denied

-a always,exit -F arch=b64 -S unlink,unlinkat -F key=file-delete

-a always,exit -F arch=b64 -S execve -F euid=0 -F auid!=0 -F auid!=-1 -F key=priv-exec

### Разбор записи SYSCALL

| Поле | Значение | Что означает |
|------|---------|-------------|
| auid | ssl | Реальный пользователь который вошёл в систему (audit user ID) |
| uid | root | Текущий UID процесса при выполнении системного вызова |
| euid | root | Эффективный UID (от чьего имени реально выполняется команда) |
| comm | cat | Имя выполняемой программы (короткое) |
| exe | /usr/bin/cat | Полный путь к исполняемому файлу |

## 3. Расследование инцидента

### Хронология действий bob

| Время | Действие | Результат | Источник |
|-------|---------|-----------|----------|
| 22:47 | cat /etc/shadow | Отказ в доступе | ausearch -k access-denied |
| 22:47 | ls /root | Отказ в доступе | ausearch -k access-denied |
| 22:47 | cat /etc/passwd | Успешно | ausearch -k auth-files |
| 22:47 | touch /tmp/totally-not-malware.sh | Успешно | ausearch --start recent |
| 22:47 | cat /etc/sudoers | Отказ в доступе | ausearch -k access-denied |

### Как auditd помог расследованию

auditd зафиксировал все системные вызовы на уровне ядра — включая отказы
в доступе. Через ausearch удалось найти конкретные действия bob с точным
временем, именем файла и результатом операции.

### Чем auditd лучше journalctl для задач аудита безопасности

journalctl фиксирует только то, что приложения сами решили записать в лог.
auditd работает на уровне ядра и перехватывает системные вызовы — приложение
не может скрыть свои действия. Также auditd фиксирует отказы в доступе,
которые journalctl не видит.
